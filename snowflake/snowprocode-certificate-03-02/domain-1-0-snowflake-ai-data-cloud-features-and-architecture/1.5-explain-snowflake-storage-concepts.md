# Snowflake Storage & Table Concepts (SnowPro Core)

---

## 🔹 Micro-Partitions

- Snowflake automatically stores table data in **micro-partitions**
- Size: ~**50 MB – 500 MB (compressed)**
- Columnar storage format

### Key Features:
- Automatically created and managed (no user control)
- Metadata stored for each micro-partition:
  - Min/Max values per column
  - Number of distinct values
- Enables **partition pruning** → faster query performance

### Benefits:
- No need for manual partitioning
- Efficient data scanning
- Improved query performance

---

## 🔹 Data Clustering

- Clustering = organizing data in micro-partitions based on specific columns

### Purpose:
- Improve **partition pruning**
- Reduce amount of data scanned

### Clustering Keys:
- Defined on large tables
- Example:
```sql
  CLUSTER BY (column_name)
```

  ### 🔹 When to Use Clustering

- Large tables with frequent filtering on specific columns  
- Columns with high cardinality  

### Notes:
- Snowflake automatically reclusters (if enabled)  
- Reclustering consumes compute resources

## 🔹 Table Types

### 1. Permanent Tables
- Default table type  
- Data persists until explicitly dropped  

**Features:**
- Time Travel: 0–90 days  
- Fail-safe: 7 days (fixed)  

---

### 2. Temporary Tables
- Exist only within a session  

**Features:**
- Automatically dropped at end of session  
- Not visible to other users  
- No Fail-safe  

---

### 3. Transient Tables
- Similar to permanent but reduced data protection  

**Features:**
- Time Travel: 0–1 day  
- No Fail-safe  
- Lower storage cost  

---

### 4. Apache Iceberg™ Tables
- Open table format supported by Snowflake  

**Features:**
- Store metadata externally  
- Interoperable with other engines (e.g., Spark)  
- Supports schema evolution  

---

### 5. External Tables
- Data stored outside Snowflake (e.g., S3, Azure Blob)  

**Features:**
- Query data without loading into Snowflake  
- Metadata maintained in Snowflake  
- Supports semi-structured data (JSON, Parquet)  

---

### 6. Dynamic Tables
- Automatically updated tables based on queries  

**Features:**
- Used for data pipelines  
- Incremental processing  
- Defined using target lag  

**Example:**
```sql
CREATE DYNAMIC TABLE my_table
TARGET_LAG = '1 minute'
AS
SELECT * FROM source_table;
```

## 🔹 View Types

### 1. Standard View
- Logical layer on top of tables  

**Features:**
- Stores only query definition  
- No data stored  
- Always reflects latest data  

---

### 2. Materialized View
- Stores precomputed results  

**Features:**
- Improves query performance  
- Automatically refreshed  
- Consumes storage  

---

### 3. Secure View
- Protects underlying data  

**Features:**
- Hides underlying table structure  
- Prevents data leakage via query optimization  
- Used for secure data sharing  

---

## 🔹 Key Exam Takeaways 🚨

- Micro-partitions enable **automatic partitioning & pruning**  
- Clustering improves performance for **large tables**  

### Table Differences

| Type       | Time Travel | Fail-safe | Scope        |
|------------|------------|-----------|--------------|
| Permanent  | 0–90 days  | 7 days    | Global       |
| Temporary  | Session    | No        | Session only |
| Transient  | 0–1 day    | No        | Global       |

---

### Important Concepts

- Materialized Views = **performance optimization**  
- Secure Views = **data security**  
- External Tables = **query without loading data**  
- Dynamic Tables = **automated pipelines**
