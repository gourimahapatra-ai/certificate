# Snowflake Objects & Concepts – Detailed Guide

## 1. Organization & Account Objects

### Organization
- Top-level entity in Snowflake.
- Represents a company or global deployment.
- Can contain multiple accounts across regions/clouds.
- Used for centralized billing and governance.

### Account
- A Snowflake deployment within a cloud region.
- Contains all data objects, users, roles, and warehouses.
- Each account is isolated but can share data via secure sharing.

---

## 2. Database Objects

### Database
- Logical container for schemas.
- Used to organize data at a high level.
- Example: `SALES_DB`, `HR_DB`.

---

### Schema
- Sub-container inside a database.
- Organizes tables, views, and other objects.
- Example: `PUBLIC`, `STAGING`, `ANALYTICS`.

---

### Tables
- Store structured data in rows and columns.
- Types:
  - Permanent
  - Temporary
  - Transient
- Support ACID transactions and Time Travel.

---

### Views
- Virtual tables based on SQL queries.
- Types:
  - Standard View
  - Secure View (for data sharing)
  - Materialized View (precomputed for performance)

---

### Stages
- Storage locations for loading/unloading data.
- Types:
  - Internal Stage (Snowflake-managed)
  - External Stage (S3, Azure Blob, GCS)
- Used with `COPY INTO` command.

---

### File Formats
- Define structure of input/output files.
- Supported formats:
  - CSV
  - JSON
  - Avro
  - Parquet
  - ORC
  - XML
- Used during data loading/unloading.

---

### User-Defined Functions (UDFs)
- Custom reusable functions created by users.
- Languages:
  - SQL
  - JavaScript
  - Python (Snowpark)
- Used to encapsulate business logic.

---

### Stored Procedures
- Used for procedural logic and workflows.
- Languages:
  - JavaScript
  - Python (Snowpark)
  - SQL (limited)
- Useful for automation and orchestration.

---

### Pipes
- Enable continuous data ingestion.
- Built on Snowpipe (serverless ingestion service).
- Automatically loads new files from stages into tables.

---

### Shares
- Allow secure data sharing across Snowflake accounts.
- No data duplication required.
- Types:
  - Provider Share
  - Consumer Share

---

### Sequences
- Generate unique numeric values.
- Commonly used for surrogate keys.
- Example: Auto-increment IDs.

---

### ML Models
- Machine learning models built using Snowpark ML.
- Enable training and inference inside Snowflake.
- Support in-database analytics and AI workflows.

---

### Applications
- Native applications built within Snowflake.
- Distributed via Snowflake Marketplace.
- Combine data + logic into shareable solutions.

---

## 3. Session & Context Variables

### Session Variables
- Temporary variables valid within a session.
- Defined using:
  ```sql
  SET var_name = value;

  ```

  Used in queries and scripts.

---

## Context Variables
- System-defined session information.

### Examples:
- `CURRENT_USER`
- `CURRENT_ROLE`
- `CURRENT_DATABASE`

- Used for dynamic query behavior.

---

## 4. Parameter Hierarchy

Snowflake parameters control system behavior at multiple levels:

### Levels (Hierarchy)
- Account Level
- User Level
- Session Level
- Object Level (Database, Schema, Table, etc.)

---

## 5. Parameter Precedence

Determines which parameter value is applied when defined at multiple levels.

### Order of Precedence (Highest → Lowest)
1. Session Level
2. User Level
3. Account Level
4. Object Level (in some cases)

---

## Example

If `TIMEZONE` is set as:

- Account Level → UTC  
- User Level → PST  
- Session Level → IST  

**Final Applied Value → IST (Session level takes precedence)**

---

## Summary

- Snowflake follows a layered structure:  
  **Organization → Account → Database → Schema → Objects**

- Data ingestion uses:  
  **Stages + File Formats + Pipes**

- Logic implementation uses:  
  **UDFs + Stored Procedures**

- Data sharing is enabled via:  
  **Shares**

- System behavior is controlled using:  
  **Parameters (Hierarchy & Precedence)**
