# Snowflake Virtual Warehouses – Configuration Guide (SnowPro Core)

## 1. What is a Virtual Warehouse?

A **Virtual Warehouse (VW)** in Snowflake is the **compute layer** responsible for:

- Running SQL queries  
- Loading/unloading data  
- Performing transformations  

It is a **cluster of compute resources (CPU, memory, I/O)** that can be started, stopped, resized, and scaled independently.

---

## 1.4 Configure Virtual Warehouses

### 🔹 Warehouse Types

#### 1. Standard Warehouse
- Default warehouse type  
- Uses **T-shirt sizing (X-Small → 6X-Large)**  
- Supports:
  - Scaling up (resize)
  - Scaling out (multi-cluster)
- Used for **general workloads**

**Gen 1 vs Gen 2**
- Gen 1: Traditional compute infrastructure  
- Gen 2: Improved performance & efficiency (same sizing model but better hardware)

---

#### 2. Snowpark-Optimized Warehouse
- Designed for:
  - Snowpark workloads (Python/Java/Scala)
  - ML / Data Science  
- Provides:
  - More memory per node  
  - Better performance for large data processing  

---

#### 3. Default Warehouse for Notebooks
- Used by Snowflake Notebooks  
- Automatically assigned compute  
- ⚠️ Not heavily tested in SnowPro Core (until GA)

---

### 🔹 Scaling Policies (Multi-cluster Warehouses)

Used when **MIN_CLUSTER ≠ MAX_CLUSTER (Auto-scale mode)**

#### 1. Standard Policy (Default)
- Prioritizes performance  
- Quickly adds clusters to avoid query queuing  
- Best for:
  - User-facing workloads  
  - BI dashboards  

#### 2. Economy Policy
- Prioritizes cost savings  
- Adds clusters more conservatively  
- Best for:
  - Batch workloads  
  - ETL jobs  

👉 Scaling policies control how clusters are started/stopped automatically

---

### 🔹 Warehouse Scaling Concepts

#### Scaling Up (Vertical Scaling)
- Increase warehouse size (e.g., Small → Large)  
- Improves:
  - Query performance  
  - Complex query execution  

#### Scaling Out (Horizontal Scaling)
- Add more clusters (multi-cluster)  
- Improves:
  - Concurrency  
  - Reduces queuing  

👉 Snowflake supports both approaches

---

### 🔹 Warehouse Configuration by Use Case

#### 1. Ad-hoc Queries
- Small to Medium warehouse  
- Single cluster  
- Auto-suspend enabled  
- Focus: **cost efficiency**

---

#### 2. Data Loading (ETL / ELT)
- Medium to Large warehouse  
- Depends on:
  - File size  
  - Number of files  
- Can tolerate some queuing  
- Focus: **throughput over latency**

---

#### 3. BI & Reporting
- Medium to Large warehouse  
- Multi-cluster enabled  
- Scaling policy: **Standard**  
- Focus: **low latency & concurrency**

---

### 🔹 Best Practices (Exam Important 🚨)

#### 1. Sizing (Up / Down)
- Start small → increase gradually  
- Larger warehouses:
  - Execute faster  
  - Cost more per second  
- Snowflake billing is **per-second**, so short bursts are efficient  

---

#### 2. Scaling (In / Out)

| Type       | When to Use      |
|------------|------------------|
| Scale Up   | Complex queries  |
| Scale Out  | High concurrency |

- Use **multi-cluster** for concurrent users  
- Use **resize** for heavy queries  

---

#### 3. Auto-Suspend & Auto-Resume
- Auto-suspend after inactivity  
- Saves cost significantly  
- Auto-resume when query arrives  

---

#### 4. Workload Management
Separate warehouses for:
- Different teams  
- Different workloads (ETL vs BI)

**Benefits:**
- No resource contention  
- Better cost tracking  

---

#### 5. High Concurrency Handling
- Use multi-cluster warehouses  
- Configure:
  - MIN_CLUSTER_COUNT  
  - MAX_CLUSTER_COUNT  

---

#### 6. Complex Queries
- Use larger warehouse size  
- Scaling out does **NOT** speed up a single query  
- Scaling up does  

---

### 🔹 Key Exam Takeaways

- Virtual Warehouse = **Compute layer**

**Two scaling methods:**
- Scale Up → Performance  
- Scale Out → Concurrency  

**Scaling policies:**
- Standard → Performance  
- Economy → Cost  

- Use separate warehouses for workload isolation  

**Always enable:**
- Auto-suspend  
- Auto-resume
