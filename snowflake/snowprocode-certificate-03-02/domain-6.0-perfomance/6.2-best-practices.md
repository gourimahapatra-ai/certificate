# ❄️ Snowflake Warehouse Best Practices

Warehouses are one of the **most important components in Snowflake** that we can control and optimize for:
- 💰 Cost efficiency  
- ⚡ Performance  
- ⚙️ Feature utilization  

---

## 🏗️ 1. Enable Auto Suspend (Best Practice #1)

### ✅ Recommendation
- Always enable **AUTO_SUSPEND** for warehouses
- Prevents **charges for idle compute**

---

### ⏱️ How it Works
- Warehouse automatically suspends after **X seconds of inactivity**
- Example:
  ```sql
  AUTO_SUSPEND = 60  -- suspends after 1 minute

  ## ⚠️ Exceptions (When NOT to Use AUTO_SUSPEND)
- Extremely steady workloads  
- Low-latency requirements (need instant response)  
- Frequent resume delay (1–2 seconds matters)  

---

## 💡 Important Consideration (Cache Impact)
- When warehouse suspends → cache is cleared  
- Frequent suspend/resume → loss of caching benefits  

---

## 🎯 Choosing the Right Value

### ❌ Avoid:
- Too low (< 1–2 min) → frequent restart  
- Too high (> 10 min) → idle cost  

---

## 📊 Best Practice
Match `AUTO_SUSPEND` with workload gaps  

| Workload Pattern                  | Recommended Approach                          |
|----------------------------------|----------------------------------------------|
| Frequent small gaps (e.g., 3 min) | Set slightly higher (e.g., 5–10 min)         |
| Long idle periods                | Use lower value                              |
| Continuous workload              | May disable                                  |

---

## ⚙️ Special Case
- `AUTO_SUSPEND = 0` or `NULL` → Never suspend  
- Minimum value = 1 second  

---

## 🔄 2. Enable Auto Resume

### ✅ Recommendation
- Always enable `AUTO_RESUME = TRUE`  

### 💡 Why?
- Automatically restarts warehouse when a query runs  
- No manual intervention needed  

``` sql

AUTO_RESUME = TRUE

```
## 🎯 Key Rule
- If `AUTO_SUSPEND` is enabled → `AUTO_RESUME` should also be enabled  

---

## 📊 3. Optimize Auto Suspend Based on Workload

### 🔍 Workload-Based Tuning

#### 🔹 ETL Workloads
- Runs → finishes → idle  
👉 Use **lower auto suspend**

#### 🔹 BI / Reporting
- Repeated queries + caching benefits  
👉 Use **slightly higher auto suspend**

#### 🔹 Data Science
- Irregular & experimental queries  
- Less cache reuse  
👉 Use **lower auto suspend**

---

## 🎯 Key Principle
- Always align auto suspend with **query patterns & idle gaps**

---

## 🏋️ 4. Choose the Right Warehouse Size

### ✅ Recommendation
Warehouse size depends on:
- Query complexity  
- Performance requirements  

---

### ⚡ Scaling Strategy
- Complex queries → **larger warehouse**  
- Simple queries → **smaller warehouse**

---

### 💡 Important Insight
- Larger warehouse = faster execution  
- Can reduce overall runtime cost  

---

### 🎯 Practical Approach
1. Start with slightly larger size  
2. Monitor performance  
3. Gradually downsize if sufficient  

---

### ⚖️ Trade-off

| Larger Warehouse       | Smaller Warehouse     |
|----------------------|----------------------|
| Faster queries ⚡     | Lower cost 💰        |
| Higher compute cost  | Slower performance   |

---

## 🚀 Key Takeaways
- ✅ Enable `AUTO_SUSPEND` to reduce idle cost  
- ✅ Enable `AUTO_RESUME` for seamless usage  
- 🎯 Tune suspend time based on workload patterns  
- ⚡ Balance cache usage vs cost  
- 📊 Choose warehouse size based on query complexity
