# ❄️ Snowflake Table Design Best Practices

Table design plays a key role in optimizing:
- 💰 Storage cost  
- ⚡ Query performance  
- 🛡️ Data protection features  

---

## 🗂️ 1. Choose the Right Table Type

Snowflake provides **3 table types**:
- **Permanent**
- **Transient**
- **Temporary**

Each differs in **data protection (Time Travel & Fail-safe)** and **storage cost**.

---

### 📊 Table Type Comparison

| Table Type | Time Travel | Fail-safe | Use Case |
|------------|------------|----------|----------|
| Permanent  | 1–90 days  | 7 days   | Production data |
| Transient  | 0–1 day    | ❌ No    | Staging / Dev |
| Temporary  | Session only | ❌ No  | Short-lived data |

---

### ✅ Best Practices

#### 🔹 Staging Tables
- Use **Transient Tables**
- Reason:
  - Data is **temporary & redundant**
  - Frequently truncated / reloaded
  - Avoid unnecessary **fail-safe storage cost**

---

#### 🔹 Production Tables
- Use **Permanent Tables**
- Reason:
  - Requires **full data protection**
  - Supports **Time Travel + Fail-safe**

---

#### 🔹 Development / Testing
- Use **Transient or Temporary Tables**
- Reason:
  - Frequent changes (insert/delete)
  - Data is non-critical
  - Reduces storage cost

---

## 📦 2. Use Appropriate Data Types

### ✅ Best Practice
- Always choose **correct and optimized data types**

---

### 📅 Example: Date Columns
- ❌ `VARCHAR` for dates → inefficient
- ✅ `DATE` data type

**Benefits:**
- Lower storage usage
- Enables date functions
- Better performance

---

### 🔢 Numeric Data
- Use numeric types (`NUMBER`, `INT`, etc.)
- Avoid storing numbers as strings

---

### 💡 Key Insight
> Efficient data types = **less storage + faster queries + lower cost**

---

## 📊 3. Clustering Keys (Use Carefully)

### ⚠️ Important
- Snowflake automatically manages **micro-partitions**
- Clustering is **NOT required by default**

---

### ✅ When to Use Clustering
- Very **large tables**
- Queries involve:
  - Frequent filtering
  - Joins on specific columns

---

### 🔍 How to Identify Need
- Check **Query Profile**
- If **high % time spent in Table Scan**, clustering may help

---

### 📈 Example Insight
- Table scan takes **~40%+ query time**
- 👉 Consider adding a **cluster key**

---

### 🎯 Best Practice
- Use clustering only if:
  - Table scan is significant
  - Query patterns are consistent

---

## 🧠 4. Choosing Cluster Keys

### ✅ Ideal Columns
- Frequently used in:
  - `WHERE` filters
  - `JOIN` conditions

---

### 📌 Example Scenario
- Data loaded by **transaction_date**
- Queries filtered by **region**

👉 Solution:
- Define **cluster key on `region`**

---

### ⚠️ Avoid Overuse
- Clustering adds:
  - Maintenance cost
  - Additional compute usage

---

## 🚀 Key Takeaways

- ✅ Choose table type based on **data lifecycle**
- 💰 Use **Transient/Temporary** to reduce storage cost
- 📦 Optimize **data types** for efficiency
- 📊 Use clustering **only when needed**
- 🔍 Analyze **query profile before optimization**

---

> 🎯 Goal: Design tables that balance **cost, performance, and data protection**
