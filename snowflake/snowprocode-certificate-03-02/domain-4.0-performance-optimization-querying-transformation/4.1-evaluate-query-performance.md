# 🚀 Snowflake Query Performance Tuning 
---

## 🔍 Query Performance Tuning
Focuses on optimizing query execution, reducing cost, and improving response time.

---

## 📊 Query Profile / Query Insights
- Visual tool to analyze query execution plan  
- Available in Snowflake UI (Snowsight)  
- Helps identify bottlenecks and inefficiencies  

---

### ⚠️ Common Performance Issues

#### 💾 Bytes Spilled to Storage
- Occurs when memory is insufficient and data spills to disk  
- Causes slower performance  

✅ Fix:
- Increase warehouse size  
- Optimize joins and aggregations  

---

#### 🔎 Inefficient Pruning
- Happens when Snowflake cannot skip unnecessary micro-partitions  

✅ Fix:
- Use proper filtering (WHERE clauses)  
- Cluster tables if needed  
- Avoid functions on filter columns  

---

#### 🔗 Exploding Joins
- Caused by joins producing excessive intermediate rows  
- Often due to missing or incorrect join conditions  

✅ Fix:
- Ensure correct join keys  
- Filter data before joining  
- Use appropriate join types  

---

#### ⏳ Queuing
- Queries wait due to insufficient compute resources  

✅ Fix:
- Increase warehouse size  
- Enable multi-cluster warehouse  
- Separate workloads  

---

## 🧾 SNOWFLAKE.ACCOUNT_USAGE Views

Provides historical data for monitoring and tuning.

---

### 📌 Query Attribution
- Identifies **which users/queries consume credits**  
- Helps track cost per workload  

---

### 📌 Query History
- Detailed record of executed queries  
- Includes:
  - Execution time  
  - Status  
  - Errors  
  - Resource usage  

---

## ⚙️ Workload Management Best Practices

---

### 🧩 Grouping Similar Workloads
- Separate workloads into different warehouses:
  - ETL jobs  
  - BI dashboards  
  - Ad-hoc queries  

### ✔️ Benefits:
- Avoid resource contention  
- Improve performance consistency  
- Better cost control  

---

## 🧠 Pro Tips
- Always check **Query Profile first** when tuning  
- Use **right-sized warehouses**  
- Enable **auto-suspend/resume**  
- Monitor using **ACCOUNT_USAGE views**  
- Optimize queries before scaling compute
