# 🔄 Snowflake Data Transformation Techniques – Quick Notes

---

## 📊 Using Data Types

### 🧾 Structured Data
- Fixed schema (tables with defined columns)  
- Example: sales table with columns (id, amount, date)  

```sql
SELECT id, amount
FROM sales
WHERE amount > 100;

```

**Semi-Structured Data**

- Flexible schema (JSON, Avro, XML, Parquet)
- Stored in VARIANT column

``` SQL

SELECT 
  data:name::STRING AS name,
  data:age::INT AS age
FROM users;

```

Unstructured Data
Files like images, videos, PDFs
Stored in stages and accessed via metadata

```sql
SELECT METADATA$FILENAME, METADATA$FILE_SIZE
FROM @my_stage;
```

➕ Aggregate Functions
Perform calculations on groups of rows
✔️ Common Functions:
SUM()
COUNT()
AVG()
MIN()
MAX()

```sql
SELECT region, SUM(amount) AS total_sales
FROM sales
GROUP BY region;

```

**Applying SQL for Query Optimization**

✔️ Best Practices:
- Use SELECT specific columns (avoid SELECT *)
- Apply filters early (WHERE clause)
- Use proper joins and conditions
- Avoid unnecessary subqueries
- Use clustering keys for large tables


```sql
SELECT id, amount
FROM sales
WHERE sale_date >= '2025-01-01';
```
## 🪟 Window Functions
Perform calculations across a **set of rows (window)** without collapsing results.

---

### ✔️ Common Functions:
- `ROW_NUMBER()`  
- `RANK()`  
- `DENSE_RANK()`  
- `SUM() OVER()`  
- `AVG() OVER()`  

---

### ✔️ Example:

```sql
SELECT 
  region,
  amount,
  ROW_NUMBER() OVER (PARTITION BY region ORDER BY amount DESC) AS row_num,
  RANK() OVER (PARTITION BY region ORDER BY amount DESC) AS rank,
  DENSE_RANK() OVER (PARTITION BY region ORDER BY amount DESC) AS dense_rank,
  SUM(amount) OVER (PARTITION BY region) AS total_region_sales,
  AVG(amount) OVER (PARTITION BY region) AS avg_region_sales
FROM sales;


```sql
SELECT 
  region,
  amount,
  ROW_NUMBER() OVER (PARTITION BY region ORDER BY amount DESC) AS rank
FROM sales;

```
## 🪟 Window Functions – Detailed Explanation

---

### 🔢 ROW_NUMBER()
- Assigns a **unique sequential number** to each row within a partition  
- No duplicates, even if values are the same  

#### ✔️ Behavior:
- Always generates **distinct numbers (1, 2, 3, …)**  
- Order is defined by `ORDER BY`  

#### ✔️ Example:
```sql
SELECT 
  region,
  amount,
  ROW_NUMBER() OVER (PARTITION BY region ORDER BY amount DESC) AS row_num
FROM sales;

```
### 🧠 Use Case:
- When you need a **unique row identifier**  
- Removing duplicates (keep first row)  

---

### 🏆 RANK()
- Assigns rank based on ordering, but **allows ties**  
- Same values → same rank  
- **Skips ranks** after ties  

---

### ✔️ Behavior:
- Example output: **1, 2, 2, 4** (rank 3 is skipped)

---

### ✔️ Example:
```sql
SELECT 
  employee_name,
  salary,
  RANK() OVER (ORDER BY salary DESC) AS rank
FROM employees;
```
### 🧠 Use Case:
- Leaderboards  
- Ranking with gaps when ties exist (use `RANK()`)

---

### 🥇 DENSE_RANK()
- Similar to `RANK()` but **does NOT skip ranks**  
- Same values → same rank  
- Next rank is **sequential (no gaps)**  

---

### ✔️ Behavior:
- Example output: **1, 2, 2, 3** (no gaps)

---

### ✔️ Example:
```sql
SELECT 
  employee_name,
  salary,
  DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank
FROM employees;

```

### 🧠 Use Case:
- Ranking where **continuous sequence is required**  
- Category/group rankings  

---

### 🔍 Quick Comparison

| Function      | Handles Ties | Skips Ranks | Unique Numbers |
|---------------|-------------|-------------|----------------|
| ROW_NUMBER()  | ❌ No       | ❌ No       | ✅ Yes         |
| RANK()        | ✅ Yes      | ✅ Yes      | ❌ No          |
| DENSE_RANK()  | ✅ Yes      | ❌ No       | ❌ No          |

---

### 🧠 Pro Tip
- Use **ROW_NUMBER()** for deduplication  
- Use **RANK()** for competition-style ranking  
- Use **DENSE_RANK()** when you need continuous ranking without gaps  

---

## 🧠 Pro Tips

- Use **VARIANT + JSON functions** for handling semi-structured data efficiently  
- Prefer **window functions** over complex subqueries for better readability and performance  
- Combine **aggregation and filtering** effectively to minimize data processing  
- Always **optimize queries first** before scaling compute resources (warehouse size)
